FAIL 1 — Security / Internal exception messages leaked to client

File: server_code/server_settings/service.py
Functions: get_business_profile, save_business_profile, get_email_config, save_email_config, get_payment_config, save_payment_config, get_theme_config, save_theme_config
Problem: All eight callables catch tables.TableError and return {'success': False, 'error': str(exc)} — this leaks the raw internal Anvil table error message to the client. The general Exception handler correctly returns 'An unexpected error occurred' but TableError does not.
Fix: Change all except tables.TableError return statements from return {'success': False, 'error': str(exc)} to return {'success': False, 'error': 'A data error occurred'} (or similar generic message). Keep logger.error(..., exc_info=True) so the real message is still logged server-side.

FAIL 3 — Error Handling / anvil.server.AnvilWrappedError not caught in client calls

Fix: Same as FAIL 2 — add except anvil.server.AnvilWrappedError handling alongside TimeoutError at every call site.
FAIL 4 — Error Handling / AssertionError from defensive assert is swallowed by except Exception

File: server_code/server_settings/service.py
Function: save_payment_config
Problem: The defensive assert not (_PAYMENT_SECRET_COLUMNS & set(fields)) is intended to be a hard fail on programmer error. However, AssertionError is a subclass of Exception, so it is caught by the except Exception block below it and silently returned as 'An unexpected error occurred' — it does not hard-fail as intended.
Fix: Add an explicit except AssertionError clause before the except tables.TableError clause that re-raises: except AssertionError: raise — this lets it propagate as a genuine programmer error rather than being swallowed.

Function: test_email_connection
